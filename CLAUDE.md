# 回答规则

回答用户的问题时,**必须严格基于本项目代码和文件**:

1. **优先读代码再回答**:在回答任何关于本项目的问题(实现、行为、配置、参数、文件位置、调用关系等)之前,先用 Read / Grep / Glob 等工具查阅项目源代码确认。
2. **禁止使用内置/通用知识替代项目事实**:不要凭训练知识或常见框架惯例推测本项目的实现。即使是常见库(NCCL、SimAI、htsim 等),也要以本项目实际代码为准,因为可能有自定义改动。
3. **找不到就诚实说找不到**:如果在项目里搜不到相关内容,直接告诉用户"项目里没找到 X",并说明搜过哪些路径/关键词。**不要编造、不要猜测、不要用通用知识补全**。
4. **引用证据**:回答时给出文件路径和行号(`path/to/file.cpp:123`),让用户能直接核对。
5. **区分事实与推测**:如果必须做推断,明确标注"这是推测"而不是当作事实陈述。

外部通用问题(与本项目无关的纯知识问题)不受此约束。

# 修改代码前的确认规则

当你怀疑代码存在逻辑错误(例如一致性问题、并发/并行问题、竞态、初始化顺序、边界条件等)时:

1. **不要立刻动手改**。先认真读相关上下文(调用方、被调方、相关数据结构、锁/同步原语、初始化和销毁路径),确认这个"错误"是不是真的存在。
2. **构造具体的反例或执行序列**来证明 bug 存在(例如:"线程 A 在 X 行读取时,线程 B 可能在 Y 行写入,因此……")。说不出具体触发路径的,就不算确认。
3. **检查是否有外部保证**让这个看似的 bug 不会发生(例如调用约定、单线程上下文、上层已经加锁、数据只在初始化阶段写入等)。
4. **如果不能确定**,先告诉用户你的怀疑和不确定的地方,让用户判断,而不是直接改代码。**错误的"修复"会引入新 bug,比原来的代码更糟。**
5. 仅在确认问题真实存在后,再提出修改方案,并说明该修改为什么不会破坏其他调用路径。

# 修改/新增代码前的方案确认规则

**无论当前处于什么权限模式(包括 acceptEdits、bypass permissions 等自动执行模式),只要涉及修改或新增代码,必须先把改动方案告诉用户,等用户选定风险级别后再执行。**

## Step 1:先给改动方案

在动手改代码之前,先输出一份**改动方案**,包含:
- 要改哪些文件、哪些函数/行
- 每处改动的内容(可以用伪代码或小 diff 描述,不需要完整实现)
- 改动的理由
- 可能的副作用、影响到的调用方

## Step 2:询问用户风险级别

问用户:**"请选择风险级别:低 / 中 / 高"**,然后按用户选择执行:

- **低风险**:直接调用 Edit / Write 执行改动,不额外审阅。
- **中风险**:执行完后,**自己再读一遍改过的代码**,主动复核:正确性、一致性、并发/并行安全、边界条件、是否有更简洁写法、是否和现有代码重复。把复核结果告诉用户。
- **高风险**:执行完后,**启动一个新的 Agent**(`subagent_type: general-purpose` 或更合适的专用 agent)独立审阅。prompt 要包含:
  - 改动目标和改动点(文件路径+行号)
  - 相关上下文文件路径(让 agent 自己读)
  - 明确请它检查:正确性、一致性、并发/并行安全、边界条件、更简洁的实现、是否重复造轮子
  - 把 agent 的审阅结果**如实**转述给用户,不要因为自己觉得对就忽略 agent 的疑问。

## 例外

**纯小改动**(改常量、日志字符串、变量名、删几行无逻辑变化的代码)可以直接做,不必先报方案、不必问风险级别。判断标准:是否引入新的执行路径或新的状态。
